# Antigravity Agent Guide

This document defines what Antigravity is, how to invoke it, how it receives
and executes missions, and how to handle errors and completions.

---

## What Is Antigravity?

Antigravity is the AI-powered implementation layer of the Rudie Orchestrator
Platform. It is the only agent authorised to write, modify, or delete code and
configuration files in the repository.

Antigravity receives structured mission files generated by Claude and executes
them one at a time or in concurrency-safe parallel batches. It does not make
architectural decisions. It does not modify missions. It implements exactly what
the mission specifies.

In practical terms, Antigravity is an AI coding agent (e.g. a Claude-powered
coding context, a Cursor agent run, or a similar agentic coding tool) that is
invoked with a mission file as its primary context.

---

## What Antigravity May Do

- Create new files
- Modify existing files explicitly listed in the mission's `FILES OWNED` field
- Run database migrations after human approval
- Install dependencies declared in the mission
- Write and run tests for the implemented code
- Signal mission completion or failure

## What Antigravity Must Never Do

- Modify `docs/ai/system-architecture.md`, `process-flows.md`, `ui-pages.md`,
  or any orchestrator-generated document
- Create or modify mission files
- Make architectural decisions not specified in the mission
- Touch files not listed in `FILES OWNED` without raising a scope flag
- Deploy to any environment directly — deployment is handled by the orchestrator
  pipeline after preflight passes

## Mission File Location

All missions live in:

```
docs/ai/missions/
```

Naming convention:

```
MISSION-[ID]-[short-slug].md
```

Examples:

```
docs/ai/missions/MISSION-001-auth-service.md
docs/ai/missions/MISSION-002-user-dashboard.md
docs/ai/missions/MISSION-003-schema-add-payments.md
```

Missions are ordered by dependency. Always check the `DEPENDS ON` field before
starting a mission.

---

## How to Start an Antigravity Agent

1. Open your agentic coding tool (Cursor, Claude Code, or equivalent).
2. Load the mission file as context:

   ```
   Use docs/ai/missions/MISSION-001-auth-service.md as your task specification.
   Follow the mission exactly. Do not deviate from FILES OWNED.
   Do not modify architecture documents.
   Signal completion by updating the STATUS field in the mission file to COMPLETE.
   ```

3. Verify the mission's `DEPENDS ON` missions are in `STATUS: COMPLETE` before
   starting.

4. If you are running multiple agents in parallel, check the
   `docs/ai/agent-concurrency-protocol.md` to confirm no file ownership
   conflicts exist.

## Mission Lifecycle

```
PENDING
  │
  ▼  (human reviews and approves)
APPROVED
  │
  ▼  (agent starts)
IN PROGRESS
  │
  ├── (implementation complete, tests pass)
  ▼
COMPLETE
  │
  └── (implementation failed or blocked)
  ▼
FAILED / BLOCKED
```

When a mission reaches `FAILED` or `BLOCKED`, the agent must:
1. Update the `STATUS` field in the mission file
2. Populate the `FAILURE NOTES` field with a clear description of what went wrong
3. Stop and not attempt further changes
4. Notify the orchestrator (Claude) by flagging the mission for re-evaluation

## Handling Scope Creep

If during implementation Antigravity discovers that completing the mission
requires touching a file not listed in `FILES OWNED`:

1. Stop immediately
2. Add a `SCOPE FLAG` to the mission file:

   ```
   SCOPE FLAG: Implementation requires modifying [filename] which is not in
   FILES OWNED. Reason: [explanation]. Awaiting orchestrator approval.
   ```

3. Set `STATUS: BLOCKED`
4. Do not touch the out-of-scope file without explicit orchestrator approval

---

## Completion Checklist

Before marking a mission `COMPLETE`, the agent must verify:

- [ ] All files in `FILES OWNED` have been created or modified as specified
- [ ] All acceptance criteria in the mission are satisfied
- [ ] Unit tests written and passing (where specified)
- [ ] No files outside `FILES OWNED` were modified
- [ ] No architecture documents were modified
- [ ] No secrets, credentials, or hardcoded environment values are present
- [ ] `npm run build` passes (or equivalent)
- [ ] TypeScript types are valid (no `any` without justification)

---

## Error Recovery

If a mission partially completes before failing:

1. Document what was completed in `FAILURE NOTES`
2. Do not attempt to undo completed steps manually
3. The orchestrator will generate a corrective mission if needed

Rollback instructions (if provided in the mission's `ROLLBACK` field) are
executed only with explicit human confirmation.

## Agent Invocation Context Template

Copy this block when starting any Antigravity agent session:

```
You are an Antigravity implementation agent operating within the Rudie
Orchestrator Platform.

Your task is defined in: [MISSION FILE PATH]

Rules:
- Read and implement the mission exactly as written
- Only modify files listed in FILES OWNED
- Do not make architectural decisions — implement only
- Do not modify any docs/ai/ orchestrator documents
- If you need to touch a file not in FILES OWNED, raise a SCOPE FLAG and stop
- Signal completion by setting STATUS: COMPLETE in the mission file
- Signal failure by setting STATUS: FAILED and populating FAILURE NOTES

Begin by reading the mission file in full before taking any action.
```

---

## Parallel Agent Execution

When running multiple Antigravity agents simultaneously, you must consult
`docs/ai/agent-concurrency-protocol.md` first.

The golden rule: **no two active missions may own the same file.**

The Prisma schema file (`prisma/schema.prisma`) has exclusive ownership — only
one migration mission may be active at any time.
